  <name>Task 1: Room migration + exportSchema (BUG-01, BUG-04)</name>
  <files>app/src/main/java/hr/fipu/organizationtool/data/AppDatabase.kt</files>
  <action>
    The current AppDatabase has two problems:
    1. `exportSchema = false` prevents migration tracking.
    2. `.fallbackToDestructiveMigration()` silently destroys user data on version bump.

    Make the following changes to AppDatabase.kt:

    1. Change the @Database annotation to `exportSchema = true`:
       `@Database(entities = [Task::class], version = 3, exportSchema = true)`
       Bump version from 2 to 3 (the migration will be a no-op placeholder since the Task schema
       has not changed between v2 and v3 — this migration exists solely to establish the pattern
       and remove the destructive fallback).

    2. Add a companion object-level migration constant before `getDatabase`:
       ```
       val MIGRATION_2_3 = object : Migration(2, 3) {
           override fun migrate(database: SupportSQLiteDatabase) {
               // Schema unchanged between v2 and v3; migration establishes safe upgrade path.
           }
       }
       ```
       Required import: `androidx.room.migration.Migration` and `androidx.sqlite.db.SupportSQLiteDatabase`.

    3. In `databaseBuilder`, replace `.fallbackToDestructiveMigration()` with `.addMigrations(MIGRATION_2_3)`.

    The `room { schemaDirectory("$projectDir/schemas") }` block already exists in app/build.gradle.kts,
    so enabling exportSchema=true will immediately cause KSP to write
    `app/schemas/hr.fipu.organizationtool.data.AppDatabase/3.json` on the next build.
  </action>
  <verify>
    Run `./gradlew :app:kspDebugKotlin` (or full build). Verify:
    - Build succeeds with no Room errors.
    - File `app/schemas/hr.fipu.organizationtool.data.AppDatabase/3.json` is created.
    - `fallbackToDestructiveMigration` no longer appears in AppDatabase.kt.
  </verify>
  <done>
    AppDatabase version is 3, exportSchema=true, MIGRATION_2_3 is registered via addMigrations(),
    and the schema JSON file exists at app/schemas/.../3.json.
  </done>
</task>

<task type="auto">
  <name>Task 2: Reset session state after saveSession (BUG-03)</name>
  <files>app/src/main/java/hr/fipu/organizationtool/ui/TaskViewModel.kt</files>
  <action>
    The saveSession() function inserts tasks but never clears _brainDumpTasks or _selectedPriorityIds.
    If the user starts a new session without restarting the app, the stale lists persist and get
    re-inserted on the next saveSession() call, causing duplicate rows (for tasks with id < 0,
    which each become id=0 and Room auto-generates new primary keys — creating true duplicates).

    After the `finalTasks.forEach { repository.insertTask(it) }` line and the widget update call,
    add the following two lines inside the viewModelScope.launch block:

    ```
    _brainDumpTasks.value = emptyList()
    _selectedPriorityIds.value = emptySet()
    ```

    This ensures each saveSession() starts from a clean slate, so suggested tasks pulled into a
    new session are only inserted once per session.
  </action>
  <verify>
    Code review: confirm the two reset lines appear after the forEach loop and ZenStackWidget().updateAll()
    inside viewModelScope.launch in saveSession(). No build errors.
  </verify>
  <done>
    saveSession() resets _brainDumpTasks to emptyList() and _selectedPriorityIds to emptySet()
    after persisting, preventing stale state from being re-submitted in subsequent sessions.
  </done>
</task>
