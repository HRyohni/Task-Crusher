<objective>
Fix the two performance regressions: reduce accelerometer polling frequency in the accessibility
service, and refactor the widget to use targeted DAO queries through TaskRepository instead of
loading all tasks via a direct AppDatabase reference.

Purpose: SENSOR_DELAY_UI polls at ~60 Hz (16 ms); SENSOR_DELAY_NORMAL polls at ~5 Hz (200 ms),
dramatically reducing battery drain on always-on gesture detection (PERF-02). The widget fetching
all tasks via AppDatabase directly bypasses the repository layer and scales poorly as the task
list grows (PERF-01, PERF-03).
Output: ZenStackAccessibilityService.kt with SENSOR_DELAY_NORMAL; TaskDao.kt with
getNonPriorityTasks(limit); ZenStackWidget.kt using TaskRepository with targeted queries.
</objective>

  <action>
    **Part A — TaskDao.kt:**
    The widget needs non-priority tasks but only displays 3 of them. Currently it calls
    `getAllTasks()` and filters + slices in memory. Add a targeted query to TaskDao:

    ```kotlin
    @Query("SELECT * FROM tasks WHERE isPriority = 0 ORDER BY createdAt DESC LIMIT :limit")
    fun getNonPriorityTasks(limit: Int): Flow<List<Task>>
    ```

    Place it after the existing `getPriorityTasks()` query. The existing `getPriorityTasks()`
    query (`SELECT * FROM tasks WHERE isPriority = 1`) is already present and correct — do not
    change it.

    **Part B — TaskRepository.kt:**
    Expose the new DAO query through TaskRepository. Add after the existing `priorityTasks` property:

    ```kotlin
    fun getNonPriorityTasks(limit: Int): Flow<List<Task>> = taskDao.getNonPriorityTasks(limit)
    ```

    **Part C — ZenStackWidget.kt:**
    The widget currently does:
      `val dao = AppDatabase.getDatabase(context).taskDao()`
    and then calls `dao.getAllTasks()`, filtering results in the composition.

    Replace with repository-backed targeted queries:
    1. Replace the AppDatabase import with a TaskRepository construction:
       ```kotlin
       val repository = TaskRepository(AppDatabase.getDatabase(context).taskDao())
       ```
       Keep the `AppDatabase` import — it is still needed to get the DAO. Remove the direct `dao`
       variable.
    2. Inside `provideContent { ... }`, replace:
       ```kotlin
       val tasks by dao.getAllTasks().collectAsState(initial = emptyList())
       val priorityTasks = tasks.filter { it.isPriority }
       val otherTasks = tasks.filter { !it.isPriority }
       ```
       With:
       ```kotlin
       val priorityTasks by repository.priorityTasks.collectAsState(initial = emptyList())
       val otherTasks by repository.getNonPriorityTasks(3).collectAsState(initial = emptyList())
       ```
    3. Remove the `tasks` variable — it is no longer needed. The `completedCount` and `progress`
       calculations already use `priorityTasks` directly, so they remain correct.
    4. Remove the `otherTasks.take(3)` call in the widget layout — the DAO now enforces the limit
       at the query level. Change `otherTasks.take(3).forEach` to `otherTasks.forEach`.
  </action>
  <verify>

    4. ZenStackWidget.kt does NOT contain `dao.getAllTasks()`.
    5. `./gradlew :app:assembleDebug` succeeds.
  </verify>
